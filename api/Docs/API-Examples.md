# API Examples

## Purpose

This guide provides practical `curl` examples for testing the Shotup Cloud API during development.

## 1. Overview

Default local base URL:

```bash
BASE_URL="http://localhost:8080"
```

All JSON requests should use:

```bash
CONTENT_TYPE="Content-Type: application/json"
```

Protected routes require Bearer auth:

```bash
AUTH_HEADER="Authorization: Bearer ACCESS_TOKEN_PLACEHOLDER"
```

Use placeholder values only in documentation, scripts, and shared examples. Do not paste real production tokens, credentials, or presigned URLs into committed files.

Common placeholders used below:

```bash
PROJECT_ID="11111111-1111-1111-1111-111111111111"
SCENE_ID="22222222-2222-2222-2222-222222222222"
FRAME_ID="33333333-3333-3333-3333-333333333333"
TRACE_ID="trace-local-0001"
```

## 2. Health Check

Endpoint:

```text
GET /api/v1/health
```

Example:

```bash
curl "$BASE_URL/api/v1/health"
```

Expected response shape:

```json
{
  "success": true,
  "data": {
    "status": "ok",
    "service": "Shotup Cloud API",
    "version": "0.1.0"
  },
  "message": null
}
```

## 3. Development Login

Endpoint:

```text
POST /api/v1/auth/dev-login
```

Example:

```bash
curl -X POST "$BASE_URL/api/v1/auth/dev-login" \
  -H "$CONTENT_TYPE" \
  -d '{
    "appleUserID": "dev.local.user",
    "email": "dev@example.com",
    "displayName": "Local Developer"
  }'
```

The response contains `accessToken`, `refreshToken`, `tokenType`, `expiresIn`, and `user`.

Manually export the access token from the response:

```bash
export ACCESS_TOKEN="<paste-access-token-from-dev-login-response>"
export AUTH_HEADER="Authorization: Bearer $ACCESS_TOKEN"
```

Do not commit real tokens.

## 4. Sync

Endpoint:

```text
POST /api/v1/sync
```

Example skeleton request:

```bash
curl -X POST "$BASE_URL/api/v1/sync" \
  -H "$AUTH_HEADER" \
  -H "$CONTENT_TYPE" \
  -d '{
    "deviceID": "local-dev-device",
    "lastSyncToken": null,
    "changes": []
  }'
```

Real sync payloads are generated by the iOS sync engine. Development curl requests are useful for smoke testing authentication, routing, and empty sync behavior.

## 5. Media Exists

Endpoint:

```text
POST /api/v1/media/exists
```

Request:

```bash
curl -X POST "$BASE_URL/api/v1/media/exists" \
  -H "$AUTH_HEADER" \
  -H "$CONTENT_TYPE" \
  -d "{
    \"frameID\": \"$FRAME_ID\"
  }"
```

Example `exists=true` response:

```json
{
  "success": true,
  "data": {
    "exists": true,
    "mediaAssetID": "44444444-4444-4444-4444-444444444444",
    "objectKey": "users/aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa/projects/11111111-1111-1111-1111-111111111111/scenes/22222222-2222-2222-2222-222222222222/frames/33333333-3333-3333-3333-333333333333/original.jpg",
    "status": "uploaded"
  },
  "message": null
}
```

Example `exists=false` response:

```json
{
  "success": true,
  "data": {
    "exists": false,
    "mediaAssetID": null,
    "objectKey": null,
    "status": null
  },
  "message": null
}
```

## 6. Media Request Upload

Endpoint:

```text
POST /api/v1/media/request-upload
```

Request:

```bash
curl -X POST "$BASE_URL/api/v1/media/request-upload" \
  -H "$AUTH_HEADER" \
  -H "$CONTENT_TYPE" \
  -H "X-Trace-ID: $TRACE_ID" \
  -d "{
    \"projectID\": \"$PROJECT_ID\",
    \"sceneID\": \"$SCENE_ID\",
    \"frameID\": \"$FRAME_ID\",
    \"contentType\": \"image/jpeg\"
  }"
```

Example response:

```json
{
  "success": true,
  "data": {
    "uploadURL": "https://example-presigned-r2-put-url",
    "objectKey": "users/aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa/projects/11111111-1111-1111-1111-111111111111/scenes/22222222-2222-2222-2222-222222222222/frames/33333333-3333-3333-3333-333333333333/original.jpg",
    "expiresAt": "2026-06-30T12:15:00Z",
    "requiredHeaders": {
      "Content-Type": "image/jpeg"
    }
  },
  "message": null
}
```

The `uploadURL` is presigned and temporary. Do not log it or commit it.

Export values manually from the response for the next steps:

```bash
export UPLOAD_URL="<paste-uploadURL>"
export OBJECT_KEY="<paste-objectKey>"
```

## 7. Upload to R2

Use the presigned `uploadURL` returned by `request-upload`.

Example:

```bash
curl -X PUT "$UPLOAD_URL" \
  -H "Content-Type: image/jpeg" \
  -H "x-required-header-placeholder: value-if-returned" \
  --data-binary "@/path/to/local/original.jpg"
```

Notes:

- Include all headers returned in `requiredHeaders`.
- Use a real local JPEG file for manual testing.
- If the URL expires, call `request-upload` again.

## 8. Media Confirm Upload

Endpoint:

```text
POST /api/v1/media/confirm-upload
```

Request:

```bash
curl -X POST "$BASE_URL/api/v1/media/confirm-upload" \
  -H "$AUTH_HEADER" \
  -H "$CONTENT_TYPE" \
  -H "X-Trace-ID: $TRACE_ID" \
  -d "{
    \"objectKey\": \"$OBJECT_KEY\",
    \"size\": 245678,
    \"mimeType\": \"image/jpeg\",
    \"checksum\": \"sha256-placeholder-if-available\"
  }"
```

Example response:

```json
{
  "success": true,
  "data": {
    "success": true
  },
  "message": null
}
```

If no checksum is available, send `null`:

```json
{
  "checksum": null
}
```

## 10. Download from R2

Use the presigned `downloadURL` returned by `request-download`.

Example:

```bash
curl -L "$DOWNLOAD_URL" \
  -o "/path/to/local/downloaded-original.jpg"
```

If the URL expires, call `request-download` again.

## 11. Common Error Examples

### 401 invalid token

Cause:

- Missing, malformed, expired, or invalid JWT.

Resolution:

- Run dev login again.
- Export a fresh `ACCESS_TOKEN`.
- Confirm the `Authorization: Bearer ...` header is present.

### 403 unauthorized

Cause:

- The authenticated user does not own the requested project, scene, shot, or media asset.

Resolution:

- Use IDs created by the same authenticated user.
- Confirm local test data belongs to the current token user.

### 404 missing dependency or media

Cause:

- `Project not found`
- `Scene not found`
- `Frame not found`
- Missing pending media asset
- Missing media asset for download

Resolution:

- Sync metadata first.
- Confirm project, scene, and shot exist in the backend.
- For upload confirmation, call `request-upload` before `confirm-upload`.

### 409 media not uploaded yet

Cause:

- `request-download` was called for a media asset whose status is still `pending`.

Resolution:

- Complete `confirm-upload`.
- Retry download after media status becomes `uploaded`.

### 400 invalid content type

Cause:

- Upload or confirm used a MIME type other than `image/jpeg`.

Resolution:

- Use `contentType: "image/jpeg"` for `request-upload`.
- Use `mimeType: "image/jpeg"` for `confirm-upload`.

## 12. Full Upload Flow

Step-by-step:

1. Start the local server.

```bash
swift run api serve
```

2. Run dev login.

```bash
curl -X POST "$BASE_URL/api/v1/auth/dev-login" \
  -H "$CONTENT_TYPE" \
  -d '{
    "appleUserID": "dev.local.user",
    "email": "dev@example.com",
    "displayName": "Local Developer"
  }'
```

3. Export `ACCESS_TOKEN` from the response.

```bash
export ACCESS_TOKEN="<paste-access-token>"
export AUTH_HEADER="Authorization: Bearer $ACCESS_TOKEN"
```

4. Sync metadata first.

Real project, scene, and shot payloads should come from the iOS sync engine. The media upload endpoint requires backend project, scene, and shot rows to already exist.

5. Request upload.

```bash
curl -X POST "$BASE_URL/api/v1/media/request-upload" \
  -H "$AUTH_HEADER" \
  -H "$CONTENT_TYPE" \
  -H "X-Trace-ID: $TRACE_ID" \
  -d "{
    \"projectID\": \"$PROJECT_ID\",
    \"sceneID\": \"$SCENE_ID\",
    \"frameID\": \"$FRAME_ID\",
    \"contentType\": \"image/jpeg\"
  }"
```

6. PUT JPEG to R2.

```bash
curl -X PUT "$UPLOAD_URL" \
  -H "Content-Type: image/jpeg" \
  --data-binary "@/path/to/local/original.jpg"
```

7. Confirm upload.

```bash
curl -X POST "$BASE_URL/api/v1/media/confirm-upload" \
  -H "$AUTH_HEADER" \
  -H "$CONTENT_TYPE" \
  -H "X-Trace-ID: $TRACE_ID" \
  -d "{
    \"objectKey\": \"$OBJECT_KEY\",
    \"size\": 245678,
    \"mimeType\": \"image/jpeg\",
    \"checksum\": null
  }"
```

8. Verify with media exists.

```bash
curl -X POST "$BASE_URL/api/v1/media/exists" \
  -H "$AUTH_HEADER" \
  -H "$CONTENT_TYPE" \
  -d "{
    \"frameID\": \"$FRAME_ID\"
  }"
```

Expected: `exists: true` and `status: uploaded`.

## 13. Notes

- Do not paste real tokens into docs.
- Do not commit secrets.
- Presigned URLs expire.
- R2 credentials never go to the client.
- Use the same `X-Trace-ID` for `request-upload` and `confirm-upload` when testing one upload attempt.
- `request-download` should not be used as a media existence probe; use `/api/v1/media/exists` for reconciliation checks.
