# M4.5 — Production Hardening

## Goal

Make the existing Project / Scene / Shot synchronization engine reliable enough to become the backend contract for the iOS SQLite sync client.

## Current Completed State

- M1 Foundation
- M2 Authenticated API
- M3 Production Authentication
- M4 Synchronization Engine
- Project sync complete
- Scene sync complete
- Shot sync complete
- Bidirectional `/sync` working

## Scope

- Persistent sync tokens
- Incremental downloads
- Last Write Wins conflict handling
- Reusable dev test scripts
- Sync integration tests
- Documentation updates

## Out Of Scope

- iOS SQLite client implementation
- Media uploads
- Cloudflare R2
- Collaboration
- Web dashboard
- Full conflict UI

## Proposed Phases

### Phase 1 — Documentation

Capture the M4.5 goals, constraints, and backend sync contract before implementation begins.

### Phase 2 — Developer Test Scripts

Replace manual curl validation with reusable scripts for authentication, project sync, scene sync, shot sync, and download verification.

### Phase 3 — Persistent Sync Tokens

Persist server-issued sync tokens so clients can safely resume synchronization from a known point.

### Phase 4 — Incremental Downloads

Use the client's last sync token to return only changes that occurred after the previous sync.

### Phase 5 — Conflict Handling

Formalize the initial Last Write Wins strategy and keep conflict responses predictable for supported entities.

### Phase 6 — Integration Tests

Add tests that cover authenticated project, scene, and shot synchronization across upload and download flows.

### Phase 7 — Final Validation And Tag

Run the full validation suite, update documentation, and mark the milestone with the final tag.

## Acceptance Criteria

- `/sync` still supports projects, scenes, and shots
- Sync responses use stable persistent tokens
- Clients can request changes since last sync
- Repeated sync requests are safe
- Old manual curl tests are replaced by reusable scripts
- Tests cover project, scene, and shot sync

## Risks

- Sync token design becoming too complex
- Conflict handling expanding beyond M4.5 scope
- Breaking the current working sync contract

## Final Tag

`M4.5-Production-Hardening`
